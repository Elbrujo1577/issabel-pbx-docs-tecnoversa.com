Alcance

Este artefacto documenta paso a paso la migración completa del sistema Issabel PBX 5: generación del backup, descarga, transferencia SCP, restauración manual por consola, cambio de puerto SSH, configuración de doble autenticación 2FA y acceso web por dominio con SSL.

03 Transferir al servidor destino

Transferir el archivo al servidor destino 192.168.50.189 usando SCP desde PowerShell en Windows, luego moverlo al directorio correcto con los permisos del usuario asterisk.

Transferencia SCP desde PowerShell

# Desde PowerShell en Windows
scp issabelbackup-20260304212014-n3.tar root@192.168.50.189:/tmp/

Mover al directorio y asignar permisos

cp /tmp/issabelbackup-20260304212014-n3.tar /var/www/backup/
chown asterisk:asterisk /var/www/backup/issabelbackup-20260304212014-n3.tar
chmod 644 /var/www/backup/issabelbackup-20260304212014-n3.tar

¿Por qué manual?
Aunque el botón Restaurar está disponible, los backups de servidor externo pueden restaurar incompleto. Se usó restauración manual para garantizar integridad total.
04 Restaurar por línea de comandos

Restauración manual en el servidor destino 192.168.50.189, extrayendo y aplicando cada componente del backup de forma individual.

Nota: El backup de servidor externo no siempre funciona desde la interfaz web del destino. Siempre usar el método manual para garantizar la restauración completa.
1
Extraer el archivo
cd /var/www/backup/
tar -xf issabelbackup-20260312155043-5g.tar
2
Restaurar BD Asterisk
Contiene extensiones, troncales, IVR, colas, rutas entrantes y salientes.
mysql asterisk < asterisk.sql
3
Restaurar BD CDR
Historial completo de llamadas.
mysql asteriskcdrdb < asteriskcdrdb.sql
4
Restaurar archivos de configuración
Restaura todo /etc/asterisk/ (sip.conf, extensions.conf, etc.)
tar -xzf etc.asterisk.tgz -C /
5
Reiniciar servicios y verificar
systemctl restart asterisk && systemctl restart httpd
asterisk -rx "sip show peers"
05 Cambio de puerto SSH

El puerto 22 es el primero que escanean los bots en Internet. Cambiarlo a un puerto personalizado elimina la mayoría de intentos de acceso automatizados.

Crítico: Abrir una segunda sesión SSH en el nuevo puerto y confirmar acceso ANTES de cerrar la sesión actual. De lo contrario se pierde el acceso remoto.
1
Editar sshd_config
vi /etc/ssh/sshd_config

# Cambiar:
#Port 22  →  Port 2244
2
Abrir puerto en firewalld
firewall-cmd --permanent --add-port=2244/tcp
firewall-cmd --reload
3
Registrar en SELinux (obligatorio en Rocky Linux)
Sin este paso SELinux bloquea SSH en el nuevo puerto aunque el firewall esté abierto.
semanage port -a -t ssh_port_t -p tcp 2244
semanage port -l | grep ssh
→ ssh_port_t    tcp    2244, 22
4
Reiniciar y verificar
systemctl restart sshd
ss -tlnp | grep sshd
→ LISTEN  0.0.0.0:2244
5
Probar conexión en el nuevo puerto
# PowerShell / Linux / macOS
ssh -p 2244 root@192.168.50.189
6
Cerrar el puerto 22 (una vez confirmado el acceso)
firewall-cmd --permanent --remove-service=ssh
firewall-cmd --reload
06 Doble autenticación 2FA

Se configuró TOTP (Time-based One-Time Password) con Google Authenticator. Al conectarse por SSH el sistema pide: contraseña + código de 6 dígitos del móvil (cambia cada 30 seg).

1
Instalar Google Authenticator PAM
dnf install -y epel-release
dnf install -y google-authenticator
2
Configurar para el usuario root
google-authenticator

# Respuestas recomendadas:
Do you want time-based tokens?            y
# → Escanear el QR con Google Authenticator / Authy
# → Anotar los 5 scratch codes (acceso de emergencia)
Update .google_authenticator file?        y
Disallow reuse of tokens?                 y
Increase time skew window?                n
Enable rate-limiting?                     y
Guardar los scratch codes. Son los únicos códigos de acceso de emergencia si se pierde el móvil.
3
Configurar PAM — agregar al inicio de /etc/pam.d/sshd
vi /etc/pam.d/sshd

# Agregar en la primera línea:
auth required pam_google_authenticator.so
4
Habilitar en sshd_config
ChallengeResponseAuthentication yes
AuthenticationMethods keyboard-interactive
5
Reiniciar SSH y probar
systemctl restart sshd

# Flujo de acceso con 2FA activo:
ssh -p 2244 root@192.168.50.189
→ Password:           ████████
→ Verification code:  847291
→ Last login: ...
07 Apache + DNS

Configurar Apache para acceder a Issabel por nombre de dominio en vez de IP. HTTP redirige automáticamente a HTTPS.

1
Crear el VirtualHost
vi /etc/httpd/conf.d/issabel-dominio.conf
# Redirigir HTTP → HTTPS
<VirtualHost *:80>
    ServerName pbx.tecnoversa.local
    Redirect permanent / https://pbx.tecnoversa.local/
</VirtualHost>

# Servir con SSL
<VirtualHost *:443>
    ServerName pbx.tecnoversa.local
    DocumentRoot /var/www/html
    SSLEngine on
    SSLCertificateFile    /etc/ssl/certs/issabel.crt
    SSLCertificateKeyFile /etc/ssl/private/issabel.key
    <Directory /var/www/html>
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>
2
Generar certificado SSL autofirmado
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/issabel.key \
  -out /etc/ssl/certs/issabel.crt \
  -subj "/C=CO/ST=Antioquia/O=Tecnoversa/CN=pbx.tecnoversa.local"
3
Abrir puertos y reiniciar Apache
firewall-cmd --permanent --add-service=http --add-service=https
firewall-cmd --reload
apachectl configtest  → Syntax OK
systemctl restart httpd
4
Registro DNS (en router o servidor DNS interno)
pbx.tecnoversa.local  →  192.168.50.189

# Alternativa temporal en hosts del cliente:
# Windows: C:\Windows\System32\drivers\etc\hosts
192.168.50.189   pbx.tecnoversa.local
Acceso final: https://pbx.tecnoversa.local — HTTP redirige automáticamente a HTTPS.
